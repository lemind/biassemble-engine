## 1. Context (evidence, not opinion)

**The gap this closes:** `llm_union` (ADR-003) has shown `positive` Recall@5 anywhere from 0.562 to 0.729 across runs (documented LLM-sampling variance, `evaluations/HISTORY.md`) against a target of 0.85. **The currently promoted baseline (`baseline_2026-07-14.json`) is 0.729**, not the lower end of that historical range — that's the actual number any fine-tune attempt is measured against (`adr/004-ci-metrics-gate.md` §4). With `positive` at only N=4 scenarios, each story is worth 0.25 of recall — 0.729 → 0.85 is not a broad, diffuse improvement, it's realistically **fixing the specific bias(es) one particular currently-failing story needs**, not a general uplift. That's precisely why §4's per-bias coverage floor (not just a total volume target) matters — an unmanaged "few hundred examples" could easily miss the exact bias this one story needs while still looking like plenty of data in aggregate. ADR-003 §10 named the fix as a future fine-tune and sketched the data source: *"`biassemble-core` already pays Gemini to label every real story; those `(story → biases)` pairs are a free SFT dataset."* This ADR is that follow-up, three things have changed since ADR-003 was written, and one of them is a real problem, not a convenience:

**(a) The training-data pump ADR-003 was counting on is real but small — first-pass verification of this was wrong and got corrected before this ADR shipped.** Queried the live Supabase instance directly (`biassemble-core`'s `core` schema, 2026-07-15):

| Table | Row count | Notes |
|---|---|---|
| `runs` | 141 | `input_hash` only — no dedicated story-text column on this table |
| `retrieval_comparisons` | 32 | `final_list` (assessment-confirmed biases) populated on all 32 |
| `retrieval_comparisons.source_breakdown` (spec-007/D017's new provenance column) | 7 | most rows predate the column |
| `retrieval_comparisons.selection_strategy = 'llm_union'` | 3 | |
| `llm_calls.raw_response` | — | stores the model's *output* text, never the input story |
| `reasoning_traces.trace → bias_hypotheses[].supporting_excerpts` | **63 of 97 rows** | **verbatim quoted fragments of the source story** — see correction below |

**Correction, caught in review:** an earlier draft of this ADR checked exactly one `reasoning_traces` row, happened to land on one with `"no_trace": true` (empty), and concluded no table stores recoverable story content. That conclusion was wrong. Querying properly: **63 of 97 `reasoning_traces` rows have `bias_hypotheses[].supporting_excerpts` populated with real, verbatim quotes from the source story** (confirmed by direct inspection — one row alone contained 8 substantial excerpts spanning most of the story's narrative arc). Average excerpt count per hypothesized bias is 2.6 — this is a **fragmentary, non-contiguous reconstruction** (quoted sentences supporting a bias judgment), not the complete original text, and coverage isn't guaranteed for parts of the story no hypothesis cited. Joining on `run_id`: **28 of those 63 runs also have a `retrieval_comparisons` row with a populated `final_list`** — meaning **28 real (partial-story-fragments, confirmed-labels) pairs are constructible from existing data today**, not zero.

This changes the shape of the problem, not the bottom line: `runs.input_hash` is still a one-way hash (can't reconstruct anything from it), and the *complete* raw story genuinely isn't stored anywhere as a clean field — but 28 partial, real pairs exist to seed the real-data side, and this codebase's existing PII-sanitization discipline (`docs/decisions/013-dataset-sanitization.md`) suggests the *absence of a dedicated story field* may still be a deliberate choice even though excerpts leak through incidentally via reasoning traces. This ADR does not propose changing what `biassemble-core` persists. Whether it should start retaining full story text (with consent, for training purposes only) stays a real, open, **cross-repo, privacy-relevant decision** this ADR explicitly does not make — flagged in §9.

**(b) Even with the 28-pair correction, the count is far short of what was already the stated bar.** ADR-002 §2 set the threshold for even considering this: *"once a few hundred story→bias pairs exist."* 28 real (fragmentary-story, confirmed-label) pairs, plus 32 label-only rows with no input at all — neither reaches "a few hundred." This isn't "a bit short," it's still roughly an order of magnitude short even counting the real seed generously.

**(c) The swap mechanism is already exactly as cheap as ADR-003 promised — this part held up.** `src/llm/generator.py`'s `LLMGenerator` takes `model_repo`/`gguf_file` as constructor args, both sourced from `config.py`'s `llm_model_repo`/`llm_gguf_file` settings — a fine-tuned model ships by changing two config values and re-pointing to a new GGUF file, zero call-site changes, exactly the contract ADR-003 §10 specified. The prompt format that any fine-tune target has to match is also fixed and simple (`src/llm/prompt.py`): system message + `bias_ids: <comma-separated ids>` + story → bare JSON array of `bias_id` strings, greedy decoding (`temperature=0.0`).

**(d) Generating synthetic training examples with LLMs is already a proven pattern in this exact codebase**, not a new idea being proposed here: `scripts/generate_story_patterns.py` already uses Gemini to generate synthetic story content for the knowledge base, and `evaluations/staging/blind_spot_eval_2026-07-13.json` is 80 DeepSeek-generated stories used for eval (explicitly marked "staged, pending spot-check before promotion" — a discipline this ADR reuses in §2/T004).

**(e) The eval infrastructure to gate a fine-tuned candidate now exists and didn't before** (`specs/005-ci-metrics-gate`, merged 2026-07-14): `scripts/check_regression.py` already does per-`(group, metric)` regression checking against a promoted baseline, and `retrieval-gate.yml` already runs the full eval harness in CI. A fine-tuned GGUF is, mechanically, just another config value — the exact same gate built for spec-005 evaluates it with no new tooling.

## 2. Decision

**Bootstrap the SFT dataset synthetically first, seeded with the 28 weak-supervision pairs that already exist; treat further real production data as a slow top-up, not the primary source** — because §1(a)/(b) show the real-data path produces a real but small number of usable pairs (28, fragmentary — quoted excerpts, not complete stories), an order of magnitude short of the volume needed.

- **Story generation: genuinely different generators, not just different model names.** "Multiple LLMs" means different providers, different model families, and different prompt templates per generator — two variants of the same model family (e.g. two Gemini tiers) share far more stylistic DNA than the diversity goal needs. The codebase already has two real data points in this direction (Gemini for story-pattern content, DeepSeek for the blind-spot batch); extend that spread, don't just alternate between near-siblings. This avoids a single family's prose style becoming a spurious feature the fine-tune learns to key on instead of the actual bias pattern.
- **Labeling: one consistent, strong teacher for the synthetic bootstrap — but this is a starting policy, not a permanent one.** ADR-003 §10's plan was distillation — a small model imitating a strong one. Mixing labelers would make the target inconsistent across examples, so use one teacher (Gemini, already trusted for real-story labeling in `biassemble-core`) for this first round. But the production system this ADR is ultimately trying to approximate isn't Gemini alone — it's Gemini + vector search + the current `llm_union` model feeding into a final assessment, and will eventually include human corrections too. That combined signal is richer supervision than a single teacher model's zero-shot judgment. **Once real production data reaches sufficient volume, future SFT rounds should prefer assessment-confirmed final labels over pure Gemini-only synthetic labels** — stated explicitly here so this ADR doesn't accidentally lock the fine-tuning pipeline into "distill Gemini" as a permanent policy rather than a bootstrap.
- **Validate every label against the 38-id catalog before it enters the dataset — not optional, not implicit.** A teacher model will routinely emit human-readable names (`"Anchoring Bias"` instead of `"anchoring_bias"`), plausible-but-nonexistent ids, or malformed JSON. Training on unchecked labels doesn't just skip a few examples — it teaches the fine-tuned model to emit strings that don't parse or don't match the catalog `prompt.py`'s parser expects. Reject (don't silently coerce or drop-and-continue) any row whose `bias_ids` contains a value outside the current 38-id catalog, before that row reaches T005's spot-check.
- **Target shape matches production exactly, not a convenient approximation:** each example is `(story_text, bias_ids[])` where `bias_ids` reproduces `src/llm/prompt.py`'s exact output contract — a bare JSON array of catalog `bias_id` strings, nothing else. A fine-tune trained on a different shape (confidence scores, evidence strings, free text) would require prompt/parsing changes to use, defeating the "zero call-site changes" swap contract from ADR-003 §10.
- **Spot-check before trusting, same discipline the blind-spot batch already uses:** a human reviews a sample of generated (story, label) pairs before they enter the training set — an LLM-labeled pipeline that nobody checked is not a ground truth, it's an unverified assumption wearing a dataset's clothes.
- **Use the 28 real pairs (§1a) as a seed — but call them what they are: weak supervision, not full examples.** Reconstruct them from `reasoning_traces.trace → bias_hypotheses[].supporting_excerpts` joined to `retrieval_comparisons.final_list` via `run_id`. "Real pairs" is the wrong mental model — these are quoted *fragments* of a story plus confirmed labels, a different and noisier modality than a complete synthetic story with the same label shape, not just a lower-fidelity version of the same thing. Label them `source: "real_weak"` in the dataset (not `"real"`) so their contribution can be measured, weighted down, or excluded separately during validation rather than silently pooled with full-story examples as if equivalent.
- **Further real production data is additive on top of that seed, not required to keep going:** as `biassemble-core` accumulates more `retrieval_comparisons` rows (and if it separately decides to start retaining full story text — §9, not this ADR's call), those examples fold into future training runs. This ADR does not block on that.
- **LoRA, not full fine-tune** — matches ADR-002/003's already-established free-tier-only constraint; a 4B model's full weights aren't trainable on free hardware, a LoRA adapter is (free-tier GPU, a few hours — platform/disk tradeoffs are an implementation detail, §5).

## 4. Dataset mechanics

- **Format**: JSONL, one object per line: `{"story": "...", "bias_ids": ["anchoring_bias", "sunk_cost_fallacy"]}` — deliberately the same shape `evaluations/{group}/*.json`'s `expected_bias_ids` already uses, so the same loader conventions apply.
- **Coverage target**: spread generated stories across the same four groups the eval harness already scores (`positive`, `negative`, `edge`, `adversarial`) — a training set skewed entirely toward clean-positive stories would not close the recall gap on the harder groups the gate already tracks.
- **Per-bias floor, not just a total-volume target — and the two numbers don't cleanly agree, on purpose.** The catalog has 38 bias IDs; "a few hundred" total *stories* averages to only ~5–10 per bias if counted one-bias-per-story, and SFT at that density improves whichever biases happen to be overrepresented, not recall uniformly. A per-bias minimum around **~15 examples per in-scope bias ID** is a reasonable starting default (not a precisely-derived number — adjust it if generation cost or early results argue for higher/lower), and 38 × 15 = 570 is *larger* than "a few hundred" *stories* only if every story is single-label. In practice each story typically implicates more than one bias (real `evaluations/positive/*.json` scenarios already do — e.g. `["hot_hand_fallacy", "illusion_of_control"]`), so multi-label stories can satisfy multiple per-bias counts at once; a few hundred *multi-label* stories can plausibly clear a ~15-per-bias floor without needing 570 *stories*. Track per-bias counts during generation regardless — don't assume multi-labeling closes the gap without checking. `positive`'s current gap (§1's opening) is driven by a specific story failing on a specific bias, not a broad deficiency; an unmanaged total count can hit "a few hundred" while still leaving exactly that bias under-covered.
- **Negative examples matter for training too, not just eval** — target **≥20% of the batch labeled `bias_ids: []`** (a starting default, not a hard requirement), matching the `negative` group's role in eval. Training exclusively on positive examples would bias the fine-tune toward always naming *something*.
- **Don't generate negatives by asking for "a story with no bias" — generate genuinely mundane content instead.** An LLM asked directly for a bias-free story tends to invent one anyway (it's optimizing for "story," not "absence of a specific pattern it wasn't asked to check for"). Prompt for ordinary, low-narrative-tension content instead — news summaries, trip reports, recipes, meeting notes, routine shopping experiences — where no cognitive-bias pattern is the natural, unforced default rather than something to actively avoid.
- **Track generation diversity along more than the bias axis — domain, writing style, and emotional tone, not just per-bias counts.** The ≥15-per-bias floor above can be satisfied by, say, 500 structurally-varied *relationship-drama* stories and still be domain-narrow. This repo already has direct evidence this matters: the blind-spot batch (`evaluations/staging/blind_spot_eval_2026-07-13.json`, `HISTORY.md`) found a real in-field/out-of-field recall gap — out-of-field domains (mycology, bonsai, paleontology) scored measurably lower than in-field ones (legal, medical, financial). Generation should deliberately span both, not skew toward whatever domains feel natural to write about.
- **Volume target**: **≥300 stories** as a starting default (aligned with ADR-002's "a few hundred" bar, made concrete) to begin a first LoRA run, not a ceiling — with the per-bias floor and the ≥20% negative fraction above tracked alongside it, not instead of it.

## 5. LoRA mechanics (free-tier constraint, per ADR-002/003)

- Base weights: same family/size as the deployed cartridge (`google/gemma-3-4b-it` or the closest fine-tunable checkpoint compatible with `bartowski`'s quantization recipe) — changing base model family here would invalidate the "drop-in GGUF swap" contract.
- Tooling: a LoRA/QLoRA library that runs on free-tier T4-class GPU (e.g. `unsloth` or `peft`+`bitsandbytes`) — specific library choice is an implementation detail for whoever executes this, not a decision this ADR needs to lock in.
- **Disk, not just GPU, matters for picking a free-tier platform for the merge step — implementation detail, not locked here, but worth flagging so it isn't a surprise.** Merging a LoRA adapter into a 4B model produces a ~7–8GB bf16 checkpoint (4B params × 2 bytes). For scale: the *already-quantized* Q4_K_M GGUF this repo runs today is ~2.4GB on disk (measured directly, `~/.cache/huggingface`) — small next to the bf16 intermediate, so the real disk pressure is almost entirely the bf16 merge step, not the final GGUF. Free-tier disk quotas vary by provider and change over time (not pinned as a decision here); if the merge step is tight on whatever's available, quantize directly from the merged checkpoint without a separate full bf16 save, rather than assuming headroom.
- Output: merge the adapter back into the base weights, then quantize with `llama.cpp`'s own conversion tooling to `Q4_K_M` GGUF — matching the exact quant format the runtime (`llama-cpp-python`) already loads, so no other runtime code changes.
- Determinism for eval comparability: keep `temperature=0.0` (already the config default, ADR-003 §5) when scoring the fine-tuned candidate against the eval harness, same as every other strategy.
- **Hold out a validation split from the SFT data itself — the eval harness is a test set, not a validation set, and can't do this job.** With only a few hundred examples on a 4B model, overfitting inside 2–3 epochs is a real risk; the retrieval-gate eval only runs *after* training completes, so it can catch a bad result but can't guide *when to stop*. Split a small fraction (~10–15% is a reasonable starting default, not a fixed requirement) of the SFT dataset out before training, track validation loss per epoch, stop at the plateau (or first rise) — the decision that matters is stopping on the validation signal, not the exact split size. **Keep the best checkpoint by validation loss, not whichever checkpoint the final epoch happens to produce** — plateau detection lags by definition (you only know you've passed the best point after seeing the next epoch get worse), so the last-saved checkpoint is frequently not the best one; save per-epoch and pick after the fact.
- **Record a manifest for every fine-tuned candidate — this repo already treats reproducibility as non-negotiable elsewhere (every promoted baseline is a versioned JSON, every eval run is logged in `HISTORY.md` with full config) and a fine-tune with no equivalent record breaks that discipline.** At minimum: SFT dataset version/hash (and its `real_weak`/`synthetic` composition), LoRA hyperparameters (rank, alpha, learning rate, epochs actually run vs. planned), base model commit/revision, the exact quantization command used, and the full eval result (§6) the candidate was judged on. Without this, a later "why did this model behave differently from that one" question has no way to be answered, and reproducing a specific candidate becomes guesswork.

## 6. Eval plan (the gate that decides whether this ships)

**Contamination risk, named explicitly:** `positive` is only N=4 stories (and `edge`/`adversarial` are N=2 each — the whole current harness is small, not just `positive`; see `adr/004-ci-metrics-gate.md` §4's worked examples). If synthetic generation (§4) happens to cover the same bias/domain combinations as those scenarios, the fine-tune is effectively training on the test distribution — a recall improvement in that case would be memorization, not generalization, and the gate would not catch it. Generation prompts must deliberately avoid replicating `evaluations/positive/*.json`'s specific story premises/domains, and whoever runs T003–T004 (the synthetic story-generation and labeling tasks — not T001's blind-spot promotion or T002's reconstruction of already-real pairs, neither of which generates anything) should diff generated story topics against the existing eval scenarios before locking in the training set.

**A second, easier-to-miss contamination path: the same teacher model on both sides.** `biassemble-core` already uses Gemini in its own assessment pipeline, and §2 uses Gemini as the synthetic-data teacher too. If Gemini (or Gemini-assisted judgment) is ever used to help construct or expand eval scenarios — not proposed anywhere in this ADR today, but an easy thing to reach for later — the fine-tuned model would effectively be evaluated by the same judge it was trained to imitate, inflating apparent performance without real generalization. **Rule: the teacher model used for synthetic labeling (§2) must never participate in constructing or labeling eval scenarios.** The current hand-authored eval sets and the DeepSeek-generated blind-spot batch already satisfy this by construction (neither used Gemini) — this rule exists to keep it true going forward, not to fix a present violation.

**Explicitly: never train on the blind-spot batch, before or after promotion.** Its entire value (§6 above, and the point directly below) is being a signal the training process never saw. Someone revisiting this months from now, looking for more training data, could reasonably reach for an 80-story batch sitting right there in `evaluations/` — that would quietly destroy the one held-out check this plan has. If it's ever intentionally folded into training, it must stop being used as an eval group in the same change, not both at once.

**Use the blind-spot batch as a real held-out test, not just the existing tiny per-group harness.** `evaluations/staging/blind_spot_eval_2026-07-13.json` (80 DeepSeek-generated stories, still staged/unpromoted per `HISTORY.md`) was never used to train anything and covers far more breadth than the current N=2–5-per-group harness. Promoting it into a real eval group **before** fine-tuning — so it's spot-checked and stable prior to any training happening — gives a genuinely held-out signal the tiny existing groups can't provide on their own, precisely because the training data in §4 is generated by the same people who could otherwise accidentally shape it to match the existing harness.

**Say plainly what §1 already implies: N=4 is too small to trust as a generalization estimate, gate or no gate.** Hitting `positive` Recall@5 ≥ 0.85 with N=4 means getting roughly 3.4 of 4 stories right — a result that specific is barely distinguishable from noise on a sample this size, and clearing it doesn't by itself prove the fine-tune generalizes rather than got lucky on exactly these 4 stories. The gate stays ≥0.85 (point 3 below) because it's the only concrete bar this repo has established and changing it isn't this ADR's call — but this is precisely why the blind-spot-batch cross-check two paragraphs up isn't optional polish, it's the only part of this plan that can distinguish "generalized" from "memorized the 4 stories, got lucky."

No new eval infrastructure otherwise — this is the payoff of having just built `specs/005-ci-metrics-gate`:

1. Point a local `run_evaluation.py --strategy llm_union` at the fine-tuned GGUF (via env override, not committed config, for the trial run).
2. Compare against the currently promoted baseline with `scripts/check_regression.py` — same per-`(group, metric)` tolerance/eligibility rule as every other change (`adr/004-ci-metrics-gate.md` §4).
3. **The bar to actually swap the deployed cartridge is the existing hard gate, unchanged: `positive` Recall@5 ≥ 0.85 (ADR-003 §7 SC-001), not "moving in the right direction."** Every other merge/swap decision in this repo's ADRs is a concrete numeric gate (ADR-002 §7, ADR-003 §7, ADR-004 §4's tolerance formulas) — this one is no exception. A result that improves on 0.729 but doesn't clear 0.85 is a **"keep iterating on data, don't ship"** outcome, not a judgment call to be made fresh each time. It must also not regress `adversarial`/`edge` past their own eligibility/tolerance (`adr/004-ci-metrics-gate.md` §4) and must not trade recall for precision — a fine-tune that raises recall by naming more biases indiscriminately isn't a win (the SC-006 precision-guard concept from ADR-003 §7 is the relevant check).
4. Only after that: update `config.py`'s defaults (or `space-vars.env`'s `LLM_MODEL_REPO`/`LLM_GGUF_FILE`) to point at the new GGUF — a one-line deploy config change, per ADR-003 §10's swap contract.

## 7. Task list (spec-kit format — detail lives in tasks.md)

- T001 — Spot-check and promote `evaluations/staging/blind_spot_eval_2026-07-13.json` into a real, held-out eval group, *before* any training data is generated — it must exist independent of and prior to §4's synthetic batch to serve as an uncontaminated signal (§6). Never a training-data source, before or after promotion (§6).
- T002 — Reconstruct the 28 weak-supervision pairs (§1a/§2): join `reasoning_traces.trace → bias_hypotheses[].supporting_excerpts` to `retrieval_comparisons.final_list` via `run_id`, tag them `source: "real_weak"` in the dataset — not `"real"`, they're fragments, not full stories.
- T003 — Generate a first synthetic batch: multiple LLMs, spread across positive/negative/edge/adversarial-shaped stories, tracking per-bias counts against the ~15-per-id floor (§4); target a few dozen as a pilot before scaling. Deliberately avoid replicating `evaluations/positive/*.json`'s specific story premises/domains (§6 contamination risk).
- T004 — Label the batch with one consistent teacher model, in `prompt.py`'s exact output shape. Validate every `bias_ids` entry against the 38-id catalog immediately — reject rows with an id outside it, don't coerce or silently drop just that id.
- T005 — Human spot-check a sample (mirrors the blind-spot batch's "staged, pending spot-check" discipline) — fix and regenerate if the failure rate is high, don't train on unverified labels.
- T006 — Scale to the full target volume (total + per-bias floor both satisfied) once the pilot's spot-check passes.
- T007 — Split off a validation set from the combined (T002 + T006) SFT data (§5), run the LoRA fine-tune (free-tier GPU, on the HF bf16 checkpoint, not the GGUF — §3) with per-epoch validation-loss tracking, stop at the plateau, keep the best checkpoint (not necessarily the last), merge, quantize to GGUF. Write the manifest (§5) alongside the output — dataset version/composition, LoRA hyperparameters, base model revision, quantization command, eval result.
- T008 — Evaluate per §6 against both the current harness and the newly-promoted blind-spot group; iterate on data (not on eval-gate tolerances) if it misses the ≥0.85 bar.
- T009 — Swap config, redeploy, confirm via the now-live `production-drift.yml` monitor (spec-005) that nothing regressed in production.

## 8. Out of scope (refuse these in-session)

- Changing `biassemble-core` to persist raw story text — real, valid, but a separate, cross-repo, privacy-relevant decision (§1a) that this ADR does not make.
- Full fine-tuning / changing base model family — LoRA on the existing cartridge only, per the free-tier constraint already established in ADR-002/003.
- GPU spend, paid compute, multi-replica training infra — same $0 constraint as every prior ADR in this repo.
- Prompt redesign — the fine-tune target is defined by `prompt.py`'s *current* shape; changing the prompt and the model at the same time would make it impossible to attribute a quality change to either one.
- Re-litigating `llm_union` vs. `nli_union`/`vector_only` — this ADR only improves the existing `llm_union` cartridge, it doesn't reopen ADR-002/003's strategy choice.

## 9. Consequences

- The real blocker to fine-tuning `llm_union` is not engineering time (§ mechanics above are all well-trodden, free-tier-viable steps) — it's that real training data tops out at 28 usable (fragmentary) pairs today, an order of magnitude short of "a few hundred," for a data-model reason (no dedicated story-text field, only incidental excerpts) that lives in a different repo.
- Whether `biassemble-core` should start retaining story text for training purposes is now a named, explicit open question (not a silent gap) — flagged here for whoever owns that repo's privacy posture, not decided by this ADR.
- Synthetic-first means the first fine-tuned cartridge is trained on LLM-generated stories, not real user stories — expect it to close some of the recall gap without fully matching what real production data would eventually teach; real data (if the cross-repo question above is ever resolved) stays additive on top of this, not a replacement for it.
- The CI investment from spec-005 pays for itself here directly — evaluating a fine-tune candidate is a config-pointer change plus a gate that already exists, not new tooling.
