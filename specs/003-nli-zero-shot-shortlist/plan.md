# Implementation Plan: NLI Zero-Shot Bias Shortlist

**Branch**: `003-nli-zero-shot-shortlist` | **Date**: 2026-07-06 | **Spec**: [spec.md](spec.md)

**Input**: [spec.md](spec.md) · [research.md](research.md) · [data-model.md](data-model.md) · [ADR-002](../../adr/002-nli-zero-shot-shortlist.md)

## Summary

Add zero-shot NLI classification (`deberta-v3-base-zeroshot-v2.0`) as the primary bias-selection signal, with existing vector search retained as a secondary union-boost signal. Selection moves from topic-similarity to behavioral-pattern recognition — the mechanism-appropriate fix for the `overconfidence_bias` ceiling that three rounds of content work could not break. A `SelectionStrategy` abstraction decouples model from interface; the same slot hosts the future fine-tuned model. Hard time-box: 7 days.

## Technical Context

**Language/Version**: Python 3.11, FastAPI

**Primary Dependencies**: `transformers[sentencepiece]`, `torch` (CPU build via `--index pytorch-cpu`), existing `pgvector`/`psycopg2` stack

**Storage**: No DB changes. `hypotheses/v1.yaml` is the only new artifact on disk.

**Testing**: pytest, existing eval harness (`run_evaluation.py`, `tune_threshold.py`)

**Target Platform**: CPU-only (HuggingFace Space free tier + local dev)

**Project Type**: web-service (FastAPI)

**Performance Goals**: ≤3s NLI inference for 38-pair batch on CPU; total request latency ≤3s via concurrent NLI + vector execution

**Constraints**:
- `biassemble-core` `RAG_TIMEOUT_MS` must be raised to 5000ms — named config change shipped with this spec
- NLI model loaded once at FastAPI startup (lifespan handler) — never per-request
- `SENTENCE_MODE=true` is offline/eval only — not a production config

**Scale/Scope**: 38 biases, one story per request, ~50–100 requests/day (current usage)

## Implementation Phases

### Phase 1: Foundation — SelectionStrategy abstraction + config (T1)

Goal: the interface exists, config plumbing is wired, `VectorOnlyStrategy` passes all existing tests unchanged.

1. Add env vars to `src/config.py`: `SELECTION_STRATEGY`, `NLI_MODEL`, `W_NLI`, `W_VEC`, `NLI_GATE`, `VEC_GATE`, `COMBINED_THRESHOLD`, `SENTENCE_MODE`, `HYPOTHESES_PATH`.
2. Create `src/selection/base.py` — `SelectionStrategy` Protocol.
3. Create `src/selection/vector_only.py` — `VectorOnlyStrategy`: wraps existing retriever output; populates absent biases with 0.0 for combiner use; exposes raw cosine as `retrieval_score` (backward-compat).
4. Wire `VectorOnlyStrategy` as default in `src/main.py` lifespan; inject into retriever.
5. Verify: existing tests pass, response shape unchanged when `SELECTION_STRATEGY=vector_only`.

**Gate**: `pytest` green, existing eval run reproduces Jul 3 baseline numbers.

---

### Phase 2: NLI inference module (T2)

Goal: model loads once, scores 38 biases in ≤3s, runs concurrently with vector search.

1. `uv add "transformers[sentencepiece]" torch --index pytorch-cpu`
2. Create `src/nli/classifier.py`:
   - Load `pipeline("zero-shot-classification", model=NLI_MODEL, device=-1)` in lifespan; store on app state.
   - `classify(story: str, hypotheses: list[str]) → NLIResult`: truncate to 512 tokens, run batch of 38, return entailment scores + latency + truncation flag.
   - `multi_label=True`, `hypothesis_template="{}"` — hardcoded, not configurable.
3. Wire concurrent execution in `NLIUnionStrategy`: `asyncio.gather(nli_task, vector_task)`.
4. Profile latency on a 200-word story; log result. If >3s, try batch-size tuning before any other change.

**Sanity check**: run `marcus_novatech.json` against hand-written overconfidence/sunk_cost/absurd hypotheses. Verify high/high/floor before writing all 38.

**Gate**: 38 scores returned in ≤3s on local CPU. `truncated_premise` fires correctly on a >512-token story.

---

### Phase 3: Hypotheses v1.yaml (T3)

Goal: all 38 hypotheses authored, loaded, producing sensible scores.

1. Create `hypotheses/v1.yaml` with `version: "v1"` and 38 entries.
2. Authoring order: start with the 5 known-miss bias families (`overconfidence`, `confirmation_bias` implicit, `sunk_cost`, `hot_hand_fallacy`, `availability_heuristic`) — these must work; the rest fill in the taxonomy.
3. Authoring rules (from research.md): actor-language, mechanism-shaped, tonal biases get tonal hypotheses, disambiguate related biases in the wording.
4. Create `src/nli/hypothesis_loader.py`: loads and validates the YAML; fails loudly at startup if any bias_id is missing or count ≠ 38.
5. Wire into lifespan alongside model load.

**Gate**: All 38 loaded at startup. The sanity ritual passes (anchoring/sunk_cost high, absurd near zero). No eval run yet — hypotheses are written cold from taxonomy, not from eval stories.

---

### Phase 4: Union-boost combiner (T4)

Goal: three-gate OR combiner produces correct admitted list, normalization is over full 38-bias vector.

1. Create `src/nli/combiner.py`:
   - Input: `nli_scores: dict[str, float]` (38), `vector_scores_raw: dict[str, float]` (38, 0.0 for absent).
   - Normalize vector scores: min-max over the full 38-value vector.
   - Apply three-gate union: admitted if `nli(b) ≥ NLI_GATE` OR `vec_norm(b) ≥ VEC_GATE` OR `combined(b) ≥ COMBINED_THRESHOLD`.
   - Return `CombinerOutput` with `admitted: list[str]` ordered by `combined_scores[id]` descending.
2. Unit-test the three gate paths independently (NLI-only admit, vector-only admit, combined-only admit, all-reject).

**Gate**: Unit tests pass for all three gate paths. `admitted` ordering is stable.

---

### Phase 5: Schema additions + structlog fields (T5)

Goal: response carries new metadata fields; logs carry selection signal info.

1. Extend `RetrievalMetadata` in `src/schemas.py` with optional fields from data-model.md.
2. Add structlog fields to the retrieval request log: `selection_strategy`, `nli_latency_ms`, `truncated_premise`, `hypotheses_version`.
3. Extend `BiasResult` if needed to carry per-bias `nli_score`/`vector_score`/`combined_score` for the returned set.

**Gate**: Response JSON includes new fields when `SELECTION_STRATEGY=nli_union`. `vector_only` response unchanged.

---

### Phase 6: Eval battery (T6)

Goal: T-eval-1, T-eval-2, T-eval-3 run and results recorded.

1. Extend `run_evaluation.py` with `--strategy` flag (`nli_only`, `vector_only`, `nli_union`).
2. Extend per-failure diagnostics: record `nli_scores`, `vector_scores`, `combined_scores`, which signal(s) missed.
3. **T-eval-1** (NLI-only, `W_VEC=0.0, NLI_GATE=0.80`): run first. Record results. This is the clean measurement of mechanism alone.
4. Extend `tune_threshold.py` with weight sweep: `W_NLI ∈ {0.5, 0.7, 0.9}` × `NLI_GATE` × `COMBINED_THRESHOLD`.
5. **T-eval-2** (union-boost sweep): find best config against gates (neg empty_rate ≥ 0.90, maximize positive recall).
6. **T-eval-3** (sentence-level, offline): run on best T-eval-2 config; record quality delta and 15-sentence-story latency.

**Gate**: T-eval-1 results recorded. Best T-eval-2 config identified. If positive Recall@5 ≥ 0.85 at any config → proceed to Phase 7. If not → T7 (hypothesis v2 iteration).

---

### Phase 7 (conditional): Hypothesis v2 iteration (T7)

Triggered only if T-eval-2 fails SC-001 (positive Recall@5 ≥ 0.85).

1. Identify failing biases from T-eval-2 diagnostics.
2. Write 2–3 alternative phrasings for each failing bias; score = max over phrasings.
3. Re-run T-eval-2 sweep with updated hypotheses.
4. One round only. If still failing → fallback path (biassemble-core ADR required).

**Gate**: Either gates pass, or fallback path is formally triggered.

---

### Phase 8: Assessment regression check + merge decision (T8)

1. Raise `RAG_TIMEOUT_MS=5000` in biassemble-core `.env` / Vercel config.
2. Run biassemble-core assessment eval (`pnpm eval`) against the winning engine config.
3. Verify no FP rate or evidence_grounded_rate regression vs pre-spec baseline.
4. Update ADR-002 status to `CLOSED — MERGED` or `CLOSED — PARKED` with final numbers.
5. Update this plan with actual results.

**Gate**: All SC-001–SC-005 pass at one config. ADR status updated.
