# Phase 0 Research: Fine-Tune the `llm_union` Cartridge

## Decision: The blind-spot batch's raw story text must be reconstructed from local, untracked files before promotion — this is a real prerequisite task, not a detail of the promote step

**Rationale**: Direct inspection of `evaluations/staging/blind_spot_eval_2026-07-13.json` shows it is an eval-*results* file, not scenario input — its records contain `scenario_id`, `group`, `domain`, `domain_familiarity`, `expected_bias_ids`, `retrieved_bias_ids`, `verdict`, etc., but no `story` field. `src/evaluation/evaluate.py`'s scenario loader requires a `story` field (confirmed against a real `evaluations/positive/*.json` file). This file cannot become a scored evaluation group as-is.

The raw story text exists, but only in 8 separate DeepSeek export files on local disk at `~/Downloads/deepseek_json_20260713_*.json` (`6a25e8`, `7da4ba`, `7e1b94`, `8c4f45`, `99e66a`, `ace574`, `bb12d7 (1)`, `c5c14a` — matching the `source_file` values recorded in the results file). Verified directly: all 8 files are present, and joining them to the results file by `(group, domain)` in list order is reliable — e.g. the `99e66a` batch's index 0 (`group: negative, domain: astronomy`) matches `neg_006`/astronomy in the results file exactly, and this held for every row checked.

This matters for two reasons beyond just "the join is possible": (1) these files exist only in a user's local `~/Downloads` folder, not in version control — a folder that could be cleared at any time, making this a one-time, use-it-or-lose-it reconstruction opportunity; (2) User Story 1 requires the held-out group to exist and be scored *before* any training data is generated, so this reconstruction is on the critical path, not a nice-to-have.

**Alternatives considered**: Treating "promote the blind-spot batch" as a single task that implicitly includes finding the story text — rejected because it hides a real, non-obvious dependency (external files, a non-trivial join) behind a task description that reads as "move a file," risking silent failure or a rushed, unverified join if whoever executes this doesn't know to look. Making it an explicit, separate task (`reconstruct_blind_spot_stories.py`, tasks.md) keeps the dependency visible and the join logic reviewable/testable on its own.

## Decision: Weak-supervision pair reconstruction and blind-spot story reconstruction are separate scripts, not one combined "reconstruct real data" task

**Rationale**: They pull from entirely different sources (biassemble-core's live Supabase vs. local DeepSeek export files) with different join keys (`run_id` vs. `(group, domain)` order) and different output shapes (SFT training pairs vs. scored evaluation scenarios feeding two different downstream consumers — the training dataset and the eval harness, respectively). Combining them into one script would couple two independent failure modes (a DB connectivity problem vs. a local-file join problem) for no shared benefit.

**Alternatives considered**: A single `reconstruct_real_data.py` with subcommands — rejected as unnecessary indirection for two scripts that share no code today; `validate_bias_catalog.py` is the actual shared logic between pipeline stages and is factored out for that reason, not these two.

## Decision: Dataset-assembly and catalog-validation logic is pure-stdlib and CI-testable; generation/labeling/training are not, and are not forced into a test shape

**Rationale**: Following the same reasoning `specs/005-ci-metrics-gate`'s research.md already established for `check_regression.py` — a function that only does arithmetic/set-membership on data already produced by something else is free, deterministic, and belongs in `tests/`. Story generation (LLM calls, non-deterministic), labeling (LLM calls), human spot-check (inherently manual), and LoRA training (compute-heavy, non-deterministic, needs GPU) are none of those things. Writing "tests" for them would either mock the interesting part away (asserting nothing real) or require live API/GPU access in CI (expensive, flaky, and this repo's explicit free-tier constraint argues against provisioning that in CI at all). The real validation for those steps is the existing eval gate at the end of the pipeline (`scripts/check_regression.py` against the promoted baseline) — which is exactly the tool spec-005 built for this purpose.

**Alternatives considered**: Mocking LLM calls to unit-test the generation/labeling scripts' control flow — rejected as low-value; the risk in those scripts is data-quality (does the model actually generate diverse, well-labeled stories), which a mock cannot exercise, and the spot-check step (User Story 2, FR-006) is the real check for that risk, not a unit test.

## Decision: `training/` is a new top-level directory, not a subdirectory of `scripts/`

**Rationale**: Every existing file in `scripts/` runs inside this repo's own `uv`-managed Python 3.11 environment, is invocable directly, and is either used in CI or callable the same way locally. The LoRA fine-tuning step runs on an external, notebook-based GPU platform (Colab/Kaggle) with its own dependency set (`unsloth`/`peft`/`bitsandbytes`, none of which belong in `pyproject.toml`) and is never executed by this repo's own tooling. Putting a documented-procedure-plus-output-artifacts directory inside `scripts/` would misrepresent it as another runnable script when it fundamentally isn't one.

**Alternatives considered**: A `notebooks/` directory — considered, but `training/` was chosen since the directory also holds the per-candidate manifest records (`training/manifests/`), which are structured data artifacts, not notebooks themselves; `training/` names the activity, not the medium.

## Decision: `evaluations/sft/` holds the assembled training dataset, as a sibling of (not nested inside) the scenario-group directories

**Rationale**: `src/evaluation/evaluate.py`'s `load_scenarios` walks every directory under `evaluations/` not in its `_SKIP_GROUPS` set and treats each as a scored scenario group. A training dataset living inside that same tree, even under a differently-named subdirectory, would be one missing entry in `_SKIP_GROUPS` away from being silently mis-scored as a scenario group — this repo already hit exactly this class of bug once (the `staging` group was missing from `_SKIP_GROUPS`, crashing `load_scenarios` on a results-shaped file, fixed in specs/005-ci-metrics-gate). `evaluations/sft/` is a sibling directory alongside the scenario groups for discoverability (it is still "evaluations-adjacent" data), but this decision is explicitly named here so a future contributor extending `_SKIP_GROUPS` logic checks this path.

**Alternatives considered**: Putting the SFT dataset under `training/dataset/` instead, fully outside the `evaluations/` tree — also would have worked and avoids the `_SKIP_GROUPS` risk entirely by construction. `evaluations/sft/` was chosen instead because the dataset's provenance (some rows derived from `evaluations/staging/`'s conventions, per-bias coverage checked against the same 4 scenario groups it must not overlap with) makes it more discoverable next to the evaluation data it's derived from and compared against — but this is a judgment call, not a strong requirement, and `training/dataset/` remains a reasonable alternative if `_SKIP_GROUPS` risk is judged to outweigh discoverability during implementation.

## Decision: `check_regression.py` requires a baseline re-promotion, not just a group promotion, before it will compare against `blind_spot` — verified against its actual implementation, not assumed

**Rationale**: Reading `scripts/check_regression.py`'s `compute_findings()` directly: it iterates `for group, baseline_gm in baseline_group_metrics.items()` — the *baseline's* groups, not the run's. A group present in a run but absent from the baseline it's compared against produces zero findings for that group, silently — this is deliberate (per the function's own docstring reasoning about `MissingGroupsError` going the other direction, run-missing-a-baseline-group), but it means promoting `blind_spot` as a scenario group (User Story 1) does not, by itself, make `check_regression.py` ever compare against it. A baseline must be re-promoted — the same manual, human step this repo already requires for any baseline change — with a `blind_spot` entry present, before FR-013's "including the held-out group" is actually true in practice, not just in the requirement's wording.

**Alternatives considered**: Modifying `check_regression.py` to iterate the union of run/baseline groups instead — rejected; this would silently start "comparing" a group with no historical baseline value (nothing to regress against), which is a worse failure mode than the current explicit "not compared yet" state, and modifying `check_regression.py` at all cuts against this feature's own stated goal of reusing spec-005's tooling completely unchanged.

## Decision: the fine-tune candidate's `eval_result` is captured by importing `compute_findings()` directly, not by parsing `check_regression.py`'s CLI output

**Rationale**: `check_regression.py`'s only output surface is `print_report()` — a formatted ASCII table on stdout, with no `--json` flag or other structured-output mode. `contracts/finetune-manifest-schema.md` requires `eval_result` to be a structured object. Parsing the printed table back into structured data would be fragile (coupling the manifest writer to column-alignment formatting meant for humans) and adding a `--json` mode to the script would be an unnecessary CLI surface change to a script this feature is supposed to reuse, not extend. `compute_findings()` is already a plain, importable Python function returning `list[RegressionFinding]` (a `@dataclass`) — the manifest-writing code imports it directly and serializes the dataclasses (e.g. via `dataclasses.asdict`), which needs zero changes to `check_regression.py` itself.

**Alternatives considered**: Adding `--json` to `check_regression.py` — rejected for the reason above (unnecessary CLI contract change); shelling out and regex-parsing the table — rejected as needlessly fragile when the underlying function is already importable.

## Decision: chat-template application, loss masking, and target serialization are documented as explicit training-loop requirements (`training/lora_finetune.md`), not left implicit

**Rationale**: None of the ADR, spec, or plan drafts prior to this review round mentioned these three points, despite each being a documented, common LoRA/SFT failure mode for instruction-tuned models: training on text that doesn't match the chat template the model was instruction-tuned with, training loss on prompt tokens instead of only the completion, and serializing the target label in a format the production parser (`src/llm/prompt.py`'s `_extract_json`) cannot read. All three "fail silently" in the specific sense that training completes, loss decreases, and only evaluation (or worse, production) reveals the fine-tune had no real effect or a negative one. Naming them explicitly as FR-019/FR-020/FR-021 (spec.md) and as required content for `training/lora_finetune.md` (not yet written — a Phase 2 task) makes them checkable requirements a reviewer of that document can verify against, rather than tribal knowledge assumed of whoever executes the training step.

**Alternatives considered**: Leaving these as assumed competence of whoever runs the LoRA step — rejected, consistent with this feature's own general stance (ADR-005's task list is written for "any AI session" executing it, not assumed-expert humans only) that load-bearing correctness requirements belong in the document, not in the executor's unstated background knowledge.

## Decision: Generation-provider and LoRA-library choices are left unresolved here, as implementation details

**Rationale**: ADR-005 §2/§5 explicitly declines to lock these in ("specific library choice is an implementation detail for whoever executes this, not a decision this ADR needs to lock in"). This plan carries that decision forward rather than re-opening it — Technical Context lists the *category* of tooling (LLM API clients for generation/labeling; `unsloth` or `peft`+`bitsandbytes` for LoRA) without picking a specific provider or library, consistent with the ADR's own stance.

**Alternatives considered**: Locking in specific providers now to make the plan more concrete — rejected because ADR-005 already made the deliberate call not to, and re-litigating that here would be scope creep for a plan whose job is to formalize the ADR's decisions, not add new ones.
